# Decisions

## Part 1 — Price filter bug

**Root cause:** The slider was hardcoded `max={100}` and `app/server/api.ts` treated `priceMax === 100` as a magic sentinel meaning "no upper limit" (`priceMax === 100 ? Number.MAX_SAFE_INTEGER : priceMax`). That worked when no car cost more than $100/hr, but the inventory now includes cars at $160, $170, and $220/hr — so users who dragged the slider to its max thinking "$100 is my ceiling" silently got *all* cars returned instead of being capped at their budget. The user from the second bug report ("MY BUDGET IS $100 PER HOUR…") hit this directly; the first user ("hide above $125") hit the other half — the slider literally couldn't represent a value above $100.

**Fix:** I removed the sentinel from `api.ts` (`priceMin` / `priceMax` are now passed straight through) and made the slider's upper bound data-driven. `getFilterOptions` now computes `maxHourlyRateDollars` from the actual inventory (`Math.max` over `hourly_rate_cents`, divided by 100, rounded up), and `AdditionalFilters.tsx` consumes it as the slider's `max`. The form's default `[10, 100]` upper bound and the "Reset all" disabled-check also use the dynamic value so they don't drift. Dragging to the right edge now genuinely means "include everything available," and any lower position is a real cap compared as-is.

**Should the fix be configurable?** I chose data-driven over config-driven on purpose. The bug's root cause was a hardcoded number going stale, so anchoring the bound to the inventory removes that recurrence risk without introducing a knob that nobody will remember to tune. A real config (env var, admin setting, feature flag) would only matter if product wanted to *artificially* cap the slider for marketing or tier reasons (e.g. "this page only shows cars under $200" even when a $500 supercar exists) — that's not today's requirement, and I'd promote it to config the moment a real reason appears. **Tradeoff acknowledged:** the deeper fix is probably to scrap the slider entirely in favor of two numeric inputs (or "Under $X" preset chips). Text inputs sidestep the whole "what's the slider's max?" question because users type the number they actually want, and they also fix the orthogonal "step=10 can't pick $125" problem from the first bug report. I've staged that critique in `possible_ux_improvements.md` rather than ship it now — it's a real product change, not a bug fix.

## Part 3 — Refactor + tests (in progress)

**Tests.** Added two test files using Node's built-in test runner (`node:test`) with `tsx` as the TypeScript loader — no new dependencies. New `npm test` script. Coverage:
- `app/server/discounts.test.ts` (17 tests, 5 suites): no-discount baselines, long-rental boundary (72h vs 73h) + math + sub-$10 clamping, holiday eligibility (middle/start/end/single-day/year-agnostic/per-holiday), best-discount selection in both directions, `HOLIDAYS` constant matches spec.
- `app/server/api.test.ts` (13 tests, 3 suites): `searchVehicles` edge cases (end ≤ start, invalid date, filter-excludes-everything, sticker-rate filtering), `getQuote` happy + throw paths, `getFilterOptions` shape.

30/30 passing.

**Bug found and fixed via the tests: invalid-date validation was dead code.** `parseAndValidateTimeRange` checked `start.toString() === "Invalid Date"`, but Luxon's invalid DateTime stringifies to `"Invalid DateTime"`. The branch never fired, so malformed date inputs silently produced `NaN` durations and totals downstream. Replaced with `!start.isValid || !end.isValid` (Luxon's documented API). Caught only because the API tests asserted on the throw behavior.

**Extracted the "Reset all" disabled condition** in `AdditionalFilters.tsx` into a named `isAtDefaultFilters()` helper. The inline expression had two real problems beyond readability: (1) it compared `values.make === undefined`, but `make` is typed `string[]` and is initialized to an array — that branch was always false, dead code; (2) `classification` wasn't checked at all, so the Reset button stayed enabled even when only the classification list was off-default. The new helper checks all five filter fields (price min/max, passengers, classification length, make length) against their defaults.

**Replaced string-literal `kind` tags with a `DiscountKind` enum** in `app/server/discounts.ts`. The `Discount` tagged union previously used `kind: "holiday"` / `kind: "long_rental"`, and every consumer (`VehicleListItem`, `ReviewPage`, both test files) compared against those raw strings. A typo in any one comparison would compile cleanly and silently fail at runtime. Centralizing the values as `DiscountKind.Holiday` / `DiscountKind.LongRental` means every reference goes through a named identifier the compiler tracks. The wire-format values (`"holiday"`, `"long_rental"`) are preserved by the string-enum form, so nothing serialized has changed.

**Added short one-line comments** to the non-obvious pieces of the discount, API, and search-context code — only where the *why* isn't apparent from the identifier (the strictly-between-days holiday iteration, the rate-clamp at $0, the best-discount tie-resolution rule, the sticker-rate filter semantics, the data-derived slider bound, and the ref-based context design). No comments on self-evident code.

### What we'd do differently if we weren't time-boxed

**Real state management via Redux.** The `SearchStateContext` we shipped is a quick ref-based React context that survives client-side navigation between the search list and the review page — and nothing else. Hard refresh wipes it, there's no audit trail of how the state got into its current shape, and the form's react-hook-form internals are still the source of truth (context is just a snapshot mirror). A proper Redux store with explicit actions and reducers for the date range and the price range would give us a single, inspectable source of truth, predictable transitions, and the foundation for features the current setup can't reach: shareable search URLs, saved searches per user, and coordinated state between the search and review pages instead of round-tripping through query params.

**Structured error types in `api.ts`.** Today the API throws plain `Error("BAD REQUEST: …")` / `Error("NOT_FOUND: …")` instances with HTTP-flavored string prefixes — a convention that callers can't safely branch on without parsing messages. `searchVehicles` papers over the issue by catching everything and returning `{ vehicles: [] }`, but that means a malformed-date error and a no-results-found state are indistinguishable to the UI. Typed error classes (`BadRequestError`, `NotFoundError`) would let consumers handle each case correctly, surface better messages to users, and integrate cleanly with HTTP status codes once the in-memory "server" is replaced by a real backend.
